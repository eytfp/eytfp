# 🚀 五种 Java Web 项目运行与部署全解析

**在学习Java代码审计时，我们有必要知道基础框架。**

**（1）因为我们往往要先启动项目，但是这可不是python代码有一个运行按钮直接运行，所以知道一些基础知识是很重要的**

**（2）同时如果我们直接去看源码，很有可能不知道从哪个目录去看去找审计点，所以我们有必要知道一些基础内容，明白为什么有些时候直接去找目录。**



在这里拆解传统 **Java Web、Spring MVC、Spring Boot、Spring 微服务、Spring 前后端分离**这五种项目的运行/部署步骤，并对比它们的核心区别。

---

## 1. 传统 Java Web 项目（Servlet + JSP）

### 🛠️ 运行与部署步骤

1. **开发阶段**

    - 使用 Servlet、JSP、JavaBean 编写代码，依赖手动导入 `servlet-api.jar`、`jsp-api.jar` 等包。

    - 项目结构：`src/main/java`（Java 代码）+ `src/main/webapp`（JSP、HTML、WEB-INF）。

2. **打包**

    - 手动或通过 Ant/Maven 打包成 `war` 包。

3. **部署**

    - 将 `war` 包复制到 Tomcat 的 `webapps` 目录。

    - 启动 Tomcat，自动解压并部署项目。

4. **访问**

    - 通过 `http://localhost:8080/项目名/xxx.jsp` 访问。

### ✨ 核心特点

- **强耦合**：前端（JSP）与后端（Servlet）代码混在一起，维护困难。

- **依赖管理**：手动导入 jar 包，容易出现版本冲突。

- **容器依赖**：必须依赖 Tomcat、Jetty 等 Web 容器才能运行。

- **部署繁琐**：每次更新都需要重新打包、上传、重启容器。

---

## 2. Spring MVC 项目

### 🛠️ 运行与部署步骤

1. **开发阶段**

    - 基于 Spring MVC 框架，使用 `@Controller`、`@RequestMapping` 等注解开发。

    - 需配置 `web.xml`（DispatcherServlet 入口）和 `springmvc-servlet.xml`（视图解析器、拦截器等）。

    - 通过 Maven 管理依赖，核心依赖为 `spring-webmvc`。

2. **打包**

    - 打包成 `war` 包，依赖 Maven 的 `war` 插件。

3. **部署**

    - 与传统 Java Web 相同，将 `war` 包放入 Tomcat 的 `webapps` 目录。

    - 启动 Tomcat 完成部署。

4. **访问**

    - 通过 `http://localhost:8080/项目名/控制器路径` 访问。

### ✨ 核心特点

- **松耦合**：通过 MVC 分层，控制器、业务逻辑、视图分离，代码更清晰。

- **依赖管理**：Maven 自动管理依赖，减少手动导入的问题。

- **容器依赖**：仍然需要 Web 容器（如 Tomcat）来运行。

- **配置复杂**：需要编写多个 XML 配置文件（或注解替代）。

---

## 3. Spring Boot 项目

### 🛠️ 运行与部署步骤

1. **开发阶段**

    - 使用 Spring Boot 快速搭建，通过 `@SpringBootApplication` 注解启动。

    - 自动配置大部分常用组件（如 Tomcat、Spring MVC、数据源），无需手动编写 XML。

    - 核心依赖为 `spring-boot-starter-web`。

2. **打包**

    - 打包成可执行 `jar` 包（内置 Tomcat）或 `war` 包（外部容器）。

    - 使用 Maven 命令：`mvn clean package`。

3. **部署**

    - **方式一（可执行 jar）**：直接运行 `java -jar xxx.jar`，内置 Tomcat 启动。

    - **方式二（war 包）**：部署到外部 Tomcat 容器。

4. **访问**

    - 通过 `http://localhost:8080/控制器路径` 访问（无需项目名）。

### ✨ 核心特点

- **约定大于配置**：自动配置，大幅减少 XML 配置文件。

- **内置容器**：默认内置 Tomcat，无需单独安装 Web 容器。

- **独立运行**：可执行 jar 包，一键启动，部署极其简单。

- **微服务友好**：天生适合微服务架构，可快速构建独立服务。

---

## 4. Spring 微服务项目

### 🛠️ 运行与部署步骤

1. **开发阶段**

    - 基于 Spring Boot + Spring Cloud 构建，每个服务都是一个独立的 Spring Boot 项目。

    - 核心组件：Eureka/Nacos（服务注册与发现）、Feign（服务调用）、Gateway（网关）、Sentinel（限流熔断）等。

    - 每个服务负责单一业务功能，通过 API 网关对外提供统一入口。

2. **打包**

    - 每个服务打包成独立的可执行 `jar` 包。

3. **部署**

    - **本地开发**：启动注册中心、配置中心、各个微服务、网关。

    - **生产环境**：使用 Docker 容器化部署，通过 Kubernetes 或 Docker Compose 管理服务集群。

    - 服务之间通过注册中心自动发现，无需硬编码 IP 地址。

4. **访问**

    - 通过 API 网关地址 `http://网关IP:端口/服务路径` 访问。

### ✨ 核心特点

- **服务拆分**：按业务拆分为多个独立服务，降低耦合，便于扩展和维护。

- **分布式治理**：依赖 Spring Cloud 组件实现服务注册、配置管理、负载均衡、熔断降级。

- **容器化部署**：通常使用 Docker + Kubernetes 实现弹性伸缩和高可用。

- **复杂度高**：需要处理分布式事务、服务调用链路追踪、一致性等问题。

---

## 5. Spring 前后端分离项目

### 🛠️ 运行与部署步骤

1. **开发阶段**

    - **后端**：基于 Spring Boot/Spring MVC 开发 RESTful API，仅返回 JSON 数据，不渲染页面。

    - **前端**：使用 Vue、React、Angular 等框架开发单页应用（SPA），通过 Axios 等工具调用后端 API。

    - 前后端通过接口文档（如 Swagger）约定数据格式。

2. **打包**

    - **后端**：打包成可执行 `jar` 包。

    - **前端**：打包成静态文件（HTML/CSS/JS）。

3. **部署**

    - **后端**：部署方式同 Spring Boot（`java -jar` 或 Docker）。

    - **前端**：将静态文件部署到 Nginx、CDN 或对象存储（如阿里云 OSS）。

    - 通过 Nginx 配置反向代理，解决跨域问题（前端请求 → Nginx → 后端 API）。

4. **访问**

    - 访问前端地址（如 `http://前端IP:端口`），前端通过 AJAX 调用后端 API。

### ✨ 核心特点

- **完全分离**：前后端代码解耦，前端负责页面渲染，后端负责业务逻辑和数据处理。

- **并行开发**：前后端团队可根据接口文档并行开发，提高效率。

- **静态部署**：前端静态资源可通过 CDN 加速，提升访问速度。

- **跨域处理**：需要通过 Nginx 反向代理或 CORS 配置解决跨域问题。

## 📊 五种项目核心区别对比

|特性|传统 Java Web|Spring MVC|Spring Boot|Spring 微服务|Spring 前后端分离|
|---|---|---|---|---|---|
|**架构风格**|单体|单体|单体/微服务|分布式微服务|前后端分离|
|**依赖管理**|手动导入 jar|Maven|Maven/Gradle|Maven/Gradle|前后端各自管理|
|**Web 容器**|外部 Tomcat|外部 Tomcat|内置/外部|内置/外部|后端内置，前端 Nginx|
|**配置复杂度**|高（XML）|中（XML/注解）|低（自动配置）|中（Spring Cloud）|中（前后端配置）|
|**部署方式**|复制 war 包|复制 war 包|一键启动 jar|容器化集群部署|后端 jar + 前端静态文件|
|**开发效率**|低|中|高|中（多服务协调）|高（并行开发）|
|**适用场景**|小型项目|中大型单体|快速开发/微服务|大型分布式系统|互联网应用、管理系统|
